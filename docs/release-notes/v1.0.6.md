# Endura — v1.0.6 (build 35)

**Released to TestFlight:** 2026-05-06 (build 34 hotfix → build 35)
**Platforms:** iOS (App Store / TestFlight) + Android (Play Store)
**Previous shipped build:** v1.0.5 (build 33)

> **Why build 35, not 34:** Build 34 went to TestFlight on 2026-05-06 and
> the recovery modal worked end-to-end, but device testing immediately
> surfaced the obvious UX flaw: the modal asked the user to pick the
> animal all over again ("shouldn't it be the animal I picked earlier?"),
> because `study_sessions` had no column for the user's start-time pick.
> Build 35 = build 34 + a new `study_sessions.intended_animal_name`
> column, persisted at `/sessions/start`, and a one-tap "Hatch your
> {animal}" confirm screen on the modal. Same App Store version (1.0.6),
> just a corrected binary. Pulled before App Store submission.

## TL;DR

A single, focused activation feature: **"hatch on next launch."** When
the server-side reaper auto-completes a stale study session, the user
now gets to hatch the animal they earned the next time they open
Endura — instead of silently losing the celebration moment.

No timer changes, no auth changes, no behaviour change for users who
finish their sessions normally.

## Why this build

Day-1 SQL across the May 4–6 cohort (post-build-33) showed:

| User age      | Sessions | Completed by user | Auto-completed by server |
|---------------|---------:|------------------:|-------------------------:|
| `mature_user` |      149 |             89.9% |                     8.7% |
| `new_user_week1` |   260 |             87.3% |                    11.9% |
| `new_user_day1`  |   325 |             65.8% |                **32.0%** |

A third of new-user day-one sessions are getting their coins/streak
credited via the reaper but never see the egg-hatch payoff — exactly
the dopamine hit that drives day-2/day-7 retention. This release
closes that gap by surfacing those sessions on the next app open and
letting the user pick the animal they want to hatch.

(The reaper itself was added in build 33 — coins/streak/totals are
already credited at reap time. This release is purely UX-side: nothing
changes about WHEN sessions are reaped or what they're worth.)

## What's new

### "Hatch on next launch" recovery flow
- New backend endpoints:
  - `GET /me/pending-hatches` — returns auto-completed sessions from
    the last 7 days that the user hasn't yet hatched the animal for.
  - `POST /sessions/{id}/hatch-pending` — user picks the animal, we
    create the `UserAnimal` row. Coins are NOT re-credited (already
    done at reap time), so this can never double-credit.
- Cold-launch + foreground hook in `App.tsx` polls `/me/pending-hatches`
  for authenticated, onboarded users.
- New `PendingHatchModal` component overlays whatever screen the user
  lands on with a horizontal animal picker + a one-tap "Hatch" button.
  Includes a "Maybe later" dismiss that snoozes for the rest of the app
  session (won't re-prompt until the next cold launch).
- Updated `push_session_recovered` push template copy:
  - **Before:** "Your study session was saved 🌳 / 25 min of biology —
    open Endura to hatch your animal."
  - **After:** "Your egg is ready to hatch 🥚 / We saved your 25 min of
    biology. Open Endura to hatch the animal you earned."

### Behavioural guards (so we don't surprise or annoy)
- **Recency cap (7 days).** A reaped session from 30 days ago doesn't
  prompt — the user has long moved on.
- **Hatch debounce.** Once the user hatches anything (via timer or via
  the recovery modal), all auto-completed sessions older than that
  hatch drop off the pending list. Prevents the awkward "hatch a
  4-day-old session after I already hatched two animals today."
- **One prompt per app session.** If the user dismisses with "Maybe
  later," we won't re-show the same session this app open. They'll see
  it again on the next cold launch.
- **Multi-session queue.** If the user has multiple pending hatches,
  we drain them oldest-first one modal at a time so the experience
  doesn't feel like a wall of nags.

### What didn't change
- The session reaper itself — still runs on the same schedule,
  same grace period (30 min), same recency horizon (168 hr / 7 d).
- Coins, streak, totals — credited at reap time as before. The new
  hatch endpoint ONLY creates the UserAnimal row.
- Push and email notifications for reaped sessions still go out as in
  build 33; only the copy changed to point at the new flow.

- GET endpoint:
  - Empty list when no reaped sessions
  - Returns reaped session with subject + minutes
  - Excludes client-completed sessions (no auto_completed_at)

## Deployment checklist

1. **Backend deploy first** — the new GET/POST endpoints need to be live
   before users on build 34 hit them. Build 34 will degrade gracefully
   (the modal just never opens) if they're missing, but the activation
   bet only pays off once both sides are shipped.
2. **(Optional but recommended) Update the live push template body.**
   The seeder in `main.py` only inserts missing rows, so the existing
   `push_session_recovered` row keeps its old "open Endura to hatch
   your animal" copy. To roll out the new "Your egg is ready to hatch
   🥚" wording, edit the row from the admin push-templates dashboard
   OR run a one-off SQL:
   ```sql
   UPDATE push_templates
   SET title = 'Your egg is ready to hatch 🥚',
       body  = 'We saved your {minutes} min of {subject}. Open Endura to hatch the animal you earned.'
   WHERE template_key = 'push_session_recovered';
   ```
3. **EAS submit for iOS** (build 34) and `eas build` for Android
   (versionCode 15). Same submission profile as build 33; no metadata
   changes.

## Risks and rollout

- **Activation experiment.** This is a UX bet — we expect to see
  day-1 hatch counts go up among new users. Watch the SQL:

  ```sql
  SELECT
    DATE(u.created_at) AS signup_day,
    COUNT(DISTINCT u.id) AS signups,
    COUNT(DISTINCT ua.user_id) FILTER (
      WHERE ua.hatched_at - u.created_at < INTERVAL '24 hours'
    ) AS day1_hatchers
  FROM users u
  LEFT JOIN user_animals ua ON ua.user_id = u.id
  WHERE u.created_at >= '2026-05-04'
  GROUP BY 1 ORDER BY 1;
  ```

  Build 34 cohort starts the day this hits the App Store; pre-build-34
  is the baseline.

- **Push delivery.** The updated push copy goes out via the same
  reaper pathway — no separate cron job, no new send schedule. If
  push deliverability looks off, the email fallback (`session_recovered`
  template, body unchanged) still fires for token-less users.

- **No-op for existing users.** If a user has no auto-completed
  sessions in the last 7 days, the GET returns `{"pending": []}` and
  the modal never mounts — zero perceptible change for them.

- **Rollback.** Backend is additive (two new endpoints + one push
  template body change). Reverting the frontend to build 33 leaves
  the new endpoints unused but harmless. The push copy update is
  applied via `push_seeds.py` ONLY when seeding fresh — production
  template rows are unchanged unless explicitly re-seeded, so the
  copy switch is gated by an explicit ops step.
